## 一、问题背景

### 1.1 现状分析

**当前系统架构**：
```
分析完成 → 更新 slow_query_template 表 → 标记 WAITING → 定时任务发送邮件
```

**核心问题**：
1. ❌ **状态覆盖问题**：同一 SQL 多次分析时，新的报告会覆盖旧报告
2. ❌ **无法查看历史通知**：用户无法查看"之前发送了哪些邮件"
3. ❌ **缺少时间线展示**：无法看到问题的发展过程
4. ❌ **前端已有"告警历史"页面**：但基于 `alert_history` 表，不是通知历史

### 1.2 用户需求

**核心诉求**：
> "通知管理功能要能够查看具体的通知内容，也就是最后时间段内且带有支持时间线展示的多个指纹的报告"

**用户担忧**：
> "数据库里保存了所有的指纹报告，数据可能比较多，报告肯定是数据库占用内存比较大的一块"

## 三、技术方案设计

### 3.1 方案对比

| 方案 | 描述 | 优点 | 缺点 | 推荐度 |
|------|------|------|------|--------|
| **方案A**<br/>纯队列模式 | 临时队列，发送后删除 | 空间占用小 | ❌ 无法查看历史 | ⭐⭐ |
| **方案B**<br/>双表设计 | 历史摘要表 + 完整报告文件 | 支持历史查询 | 需管理文件系统 | ⭐⭐⭐⭐ |
| **方案C**<br/>分级存储 | 近期数据库 + 历史文件 | 灵活、可控 | 实现复杂 | ⭐⭐⭐ |
| **方案D**<br/>压缩存储 | 数据库 + GZIP | 简单 | 无法查询压缩内容 | ⭐⭐⭐ |
| **方案E**<br/>直接保存 | 完整报告直接入库 | ✅ 简单直接<br>✅ 先跑通功能<br>✅ 存储可控（~1GB/年） | ⚠️ 需要定期清理 | ⭐⭐⭐⭐⭐ |

### 3.2 推荐方案：方案E（直接保存完整报告）

**用户决策**：
> "那算了，还是直接保存吧每次的通知，吧完整的报告落库吧，先把功能跑通再说"

**设计思路**：
- **notification_queue（队列表）**：临时存储，发送后立即删除（成功或失败）
- **notification_history（历史表）**：永久存储完整报告，支持历史查询
- **存储策略**：直接保存完整 HTML 报告到数据库 TEXT 字段
- **清理策略**：可选实现定期清理（>90天），暂时不实现

**为什么选择方案E？**
1. ✅ **简单直接**：不需要复杂的文件系统管理
2. ✅ **先跑通功能**：核心功能优先，优化后续再做
3. ✅ **存储可控**：预估年数据量 ~1GB，完全可接受
4. ✅ **查询方便**：SQL 直接查询，不需要读文件
5. ✅ **备份简单**：数据库备份即可，不需要额外处理文件

### 4.1 notification_history（通知历史表）

```sql
CREATE TABLE notification_history (
    -- 主键
    id BIGINT AUTO_INCREMENT PRIMARY KEY,

    -- 关联信息
    batch_id VARCHAR(64) NOT NULL COMMENT '批次ID（UUID，用于标识一次通知）',
    sql_fingerprint VARCHAR(255) NOT NULL COMMENT 'SQL指纹',
    db_name VARCHAR(100) NOT NULL COMMENT '数据库名',
    table_name VARCHAR(255) COMMENT '表名',

    -- 时间信息
    analyzed_time DATETIME NOT NULL COMMENT '分析时间',
    sent_time DATETIME COMMENT '发送时间',

    -- 严重程度
    severity VARCHAR(20) NOT NULL COMMENT 'CRITICAL/WARNING/NORMAL',

    -- 统计信息
    occurrence_count BIGINT NOT NULL COMMENT '执行次数',
    avg_query_time DOUBLE COMMENT '平均耗时(秒)',
    max_query_time DOUBLE COMMENT '最大耗时(秒)',

    -- 报告内容（完整）
    sql_template TEXT COMMENT 'SQL模板',
    ai_report TEXT COMMENT 'AI分析报告（完整）',

    -- 通知信息
    notification_status VARCHAR(20) NOT NULL COMMENT 'SENT/FAILED',
    error_message TEXT COMMENT '失败原因',

    -- 元数据
    created_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    -- 索引
    INDEX idx_batch_id (batch_id),
    INDEX idx_sql_fingerprint (sql_fingerprint),
    INDEX idx_sent_time (sent_time DESC),
    INDEX idx_analyzed_time (analyzed_time DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='通知历史表';
```

### 4.3 存储策略（方案E：直接保存）

| 数据类型 | 保留时间 | 存储位置 | 预估数据量 | 说明 |
|---------|---------|----------|-----------|------|
| 临时队列 | 最多15秒 | 数据库 | ~10 MB | 发送后立即删除（成功或失败） |
| 历史报告 | 永久保存 | 数据库（完整报告） | ~1 GB/年 | 直接保存完整HTML报告 |

**存储估算**（基于实际情况）：
- 假设每天 10 条通知，每条报告平均 50KB
- 年数据量：10 条/天 × 50KB × 365 天 ≈ **182.5 MB/年**
- 考虑到索引、元数据等，预估 **~1 GB/年**

**结论**：存储成本完全可接受，不需要复杂的分级存储策略。

### 6.2 通知队列重试机制（重要）

**设计原则**：
> `notification_queue` 是临时队列表，**不是历史表**。发送后立即删除（成功或失败），不需要定期清理任务。

**重试逻辑（立即重试，写死3次）**：
```
发送邮件
    ↓
发送成功 → 立即删除队列记录 ✅
    ↓
发送失败 → 立即重试（等待5秒）
    ↓
第2次失败 → 立即重试（等待5秒）
    ↓
第3次失败 → 立即重试（等待5秒）
    ↓
第4次失败（3次重试均失败） → 删除队列记录 ❌
```

**实现细节**：
```java
// NotificationScheduler - 立即重试机制
@Scheduled(cron = "${db-doctor.notify.batch-cron:0 0 * * * ?}")
public void batchSendNotifications() {
    // 1. 查询待发送队列
    List<NotificationQueue> pendingQueues = notificationQueueRepo
        .findByStatusAndAnalyzedTimeBetween(...);

    // 2. 构建批次报告
    NotificationBatchReport report = buildBatchReportFromQueue(...);

    // 3. 发送批量通知（支持立即重试，最多3次）
    boolean sendSuccess = false;
    final int MAX_RETRY_TIMES = 3; // 写死，不需要配置

    for (int retryCount = 0; retryCount < MAX_RETRY_TIMES; retryCount++) {
        sendSuccess = notifyService.sendBatchNotification(report);

        if (sendSuccess) {
            log.info("✅ 批量通知发送成功（第 {} 次尝试）", retryCount + 1);
            break; // 发送成功，退出重试循环
        }

        // 发送失败
        log.warn("⚠️ 第 {} 次发送失败", retryCount + 1);

        // 如果还有重试机会，等待5秒后重试
        if (retryCount < MAX_RETRY_TIMES - 1) {
            log.info("🔄 5秒后将进行第 {} 次重试...", retryCount + 2);
            try {
                Thread.sleep(5000); // 等待5秒后重试
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                log.error("❌ 重试等待被中断");
                break;
            }
        }
    }

    // 4. 根据发送结果处理队列记录
    if (sendSuccess) {
        notificationQueueRepo.deleteAll(pendingQueues);
        log.info("✅ 已删除 {} 条队列记录", pendingQueues.size());
    } else {
        notificationQueueRepo.deleteAll(pendingQueues);
        log.error("❌ 已重试 {} 次，删除 {} 条队列记录",
            MAX_RETRY_TIMES, pendingQueues.size());
    }
}
```

**NotificationQueue 实体类（简化版）**：
```java
@Entity
@Table(name = "notification_queue")
public class NotificationQueue {
    private Long id;
    private String sqlFingerprint;
    private String dbName;
    private String tableName;
    private String sqlTemplate;
    private SeverityLevel severity;
    private String aiReport;
    private Double queryTime;
    private LocalDateTime analyzedTime;
    private NotificationStatus status;  // PENDING / SENT

    public enum NotificationStatus {
        PENDING,   // 待发送
        SENT       // 已发送
    }
}
```

**为什么不需要定期清理任务**？
1. ✅ **队列表是临时表**：发送后立即删除（成功或失败）
2. ✅ **失败立即重试**：不会累积大量失败记录
3. ✅ **最多15秒**：一次发送 + 3次重试，每次间隔5秒，总耗时不超过15秒
4. ❌ **不需要定期清理**：避免不必要的定时任务和数据库查询

**为什么写死3次，不需要配置**？
1. ✅ **简化配置**：减少配置项，降低维护成本
2. ✅ **合理默认值**：3次重试足以应对临时网络故障
3. ✅ **避免过度重试**：如果邮件服务宕机，立即重试无意义
4. ✅ **快速失败**：15秒后删除记录，不会占用数据库空间

**时间线示例**：
```
10:00:00 - 定时任务启动
10:00:01 - 第1次发送失败
10:00:06 - 第2次发送失败
10:00:11 - 第3次发送失败
10:00:16 - 第4次发送失败（3次重试均失败）
10:00:16 - 删除队列记录
```

## 九、性能优化建议

   - 完整报告按需加载

- 预估年数据量：~1 GB（完全可接受）

2. ✅ **先跑通功能**：核心功能优先，优化后续再做
3. ✅ **存储可控**：预估年数据量 ~1GB，完全可接受
4. ✅ **查询方便**：SQL 直接查询，不需要读文件
5. ✅ **备份简单**：数据库备份即可，不需要额外处理文件

### 11.2 关键设计原则

1. **时间线优先**：保留完整的发展过程
2. **队列立即重试机制**：发送失败立即重试 3 次（写死），每次间隔 5 秒，15 秒内快速失败
3. **临时表设计**：`notification_queue` 不需要定期清理，发送后立即删除
4. **历史表直接保存**：`notification_history` 完整报告直接入库，不搞分级存储
5. **性能优先**：索引优化、分页查询
6. **监控告警**：空间和性能双重监控（可选）
