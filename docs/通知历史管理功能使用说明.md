## 一、功能概述

### 1.1 核心价值

解决**状态覆盖问题**，保留完整的分析时间线：

```
之前的问题：
10:05 🔴 CRITICAL → 更新 Template 表
10:20 🟡 WARNING → 覆盖 Template 表
10:35 🟢 NORMAL   → 再次覆盖
❌ 用户只看到最后一次 "正常" 的通知

现在的解决方案：
10:05 🔴 CRITICAL → 插入 notification_queue
10:20 🟡 WARNING → 插入 notification_queue
10:35 🟢 NORMAL   → 插入 notification_queue
✅ 定时任务聚合时间线，用户看到完整的发展过程
```

### 1.2 技术方案

- **临时队列表** (`notification_queue`): 事件驱动，发送后立即删除
- **历史记录表** (`notification_history`): 永久存储，支持历史查询
- **立即重试机制**: 失败后立即重试3次，每次间隔5秒

## 二、数据表结构

### 2.1 notification_queue（通知队列表）

**用途**: 临时存储，发送后立即删除（成功或失败）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| sql_fingerprint | VARCHAR(255) | SQL指纹 |
| db_name | VARCHAR(100) | 数据库名 |
| table_name | VARCHAR(255) | 表名 |
| sql_template | TEXT | SQL模板 |
| severity | VARCHAR(20) | 严重程度 |
| ai_report | TEXT | AI分析报告 |
| query_time | DOUBLE | 平均耗时 |
| analyzed_time | DATETIME | 分析时间 |
| status | VARCHAR(20) | PENDING/SENT |

**生命周期**: 最长15秒（1次发送 + 3次重试）

### 2.2 notification_history（通知历史表）

**用途**: 永久存储完整通知报告

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| batch_id | VARCHAR(64) | 批次ID（UUID） |
| sql_fingerprint | VARCHAR(255) | SQL指纹 |
| db_name | VARCHAR(100) | 数据库名 |
| analyzed_time | DATETIME | 分析时间 |
| sent_time | DATETIME | 发送时间 |
| severity | VARCHAR(20) | 严重程度 |
| sql_template | TEXT | SQL模板 |
| ai_report | TEXT | AI分析报告（完整） |
| notification_status | VARCHAR(20) | SENT/FAILED |
| error_message | TEXT | 失败原因 |

**存储成本**: ~1 GB/年（完全可接受）

## 三、API 接口文档

### 3.1 查询通知历史列表

```
GET /api/notifications/history
```

**参数**:
- `page`: 页码（从0开始，默认0）
- `size`: 每页大小（默认20）
- `startDate`: 开始日期（格式：yyyy-MM-dd，可选）
- `endDate`: 结束日期（格式：yyyy-MM-dd，可选）
- `status`: 通知状态（SENT/FAILED，可选）

## 四、业务流程

### 4.1 通知发送流程

```
┌─────────────────────────────────────────┐
│ 1. 分析完成              │
│    ↓                                     │
│  更新 slow_query_template 表              │
│    ↓                                     │
│  插入 notification_queue 队列             │
│  status = PENDING, retryCount = 0        │
└─────────────────────────────────────────┘
                 ↓
┌─────────────────────────────────────────┐
│ 2. 定时任务执行（每小时）                  │
│    ↓                                     │
│  从 notification_queue 读取 PENDING 记录  │
│    ↓                                     │
│  生成批次ID (UUID)                        │
│    ↓                                     │
│  按指纹聚合（支持时间线）                  │
│    ↓                                     │
│  发送邮件（立即重试，最多3次）              │
└─────────────────────────────────────────┘
                 ↓
        ┌────────────┴────────────┐
        ↓                         ↓
   发送成功 ✅                发送失败 ❌
        ↓                         ↓
  保存到                    保存到
  notification_           notification_
  history (SENT)          history (FAILED)
        ↓                         ↓
  删除队列记录               删除队列记录
```

### 4.2 立即重试机制

```
10:00:01 - 第1次发送失败
10:00:06 - 第2次发送失败（等待5秒后重试）
10:00:11 - 第3次发送失败（等待5秒后重试）
10:00:16 - 第4次发送失败（等待5秒后重试）
10:00:16 - 删除队列记录，保存失败历史
```

**特点**:
- ✅ 立即重试（不等下次定时任务）
- ✅ 写死3次（不需要配置）
- ✅ 每次间隔5秒
- ✅ 总耗时不超过15秒

## 七、常见问题

### Q1: notification_queue 表会不会积累太多数据？

**A**: 不会。队列表是临时表，发送后立即删除（成功或失败），最长保留15秒。

### Q2: notification_history 表占用多少空间？

**A**: 预估 ~1 GB/年，完全可接受。

### Q3: 失败的通知会重试吗？

**A**: 会的。失败后立即重试3次，每次间隔5秒。3次都失败后，保存失败记录并删除队列。

### Q4: 为什么重试次数写死为3次？

**A**: 简化配置，降低维护成本。3次重试足以应对临时网络故障。

### Q5: 需要定期清理历史数据吗？

**A**: 当前阶段不需要。如果数据量增长超出预期，可以后续添加清理任务（删除90天前的数据）。
